# README_GEN.md

## Overview
The gen module generates Nix modules from lock file entries. Each package in the lock becomes a standalone Nix module with proper markers for tracking and cleanup.

## Location
`lib/gen.sh`

## Functions

### `is_derivation(pkg)`
Checks if a package is a valid Nix derivation (not just a namespace).

**Parameters:**
- `$1` - Package attribute path

**Returns:** 0 if derivation, 1 otherwise

**Implementation:** Nix eval using `tryEval` for safety

---

### `generate_modules()`
Main function: reads lock file and generates Nix modules.

**Workflow:**
1. Read all entries from lock file
2. For each entry:
   - Validate it's a derivation (skip if not)
   - Check if module already exists
   - Create module if new
3. Display progress with item counts
4. Show summary: generated | existed | skipped

**Output Format:**
```
  ▶ Generating Modules
  ─────────────────────────────────────────────────────────────
  ℹ Processing 42 lock entries
  ✓ Generated: firefox.nix
  ○ Exists: vim.nix
  ✗ Skipped non-package: some-namespace
  ...
  Summary: 15 generated | 25 existed | 2 skipped
```

**Generated Module Structure:**
```nix
{ config, pkgs, ... }:

{
  environment.systemPackages = with pkgs; [
    firefox
  ];
}

#$nixorcist$#
# NIXORCIST-ATTRPATH: firefox
```

---

### `purge_all_modules()`
Removes all nixorcist-generated modules.

**Behavior:**
1. Iterate modules directory
2. Check for NIXORCIST_MARKER in each file
3. Delete only marked files (safe cleanup)
4. Clear lock file
5. Show count of removed modules

**Usage:**
```bash
purge_all_modules
```

---

## Module Naming

Package names are transformed to safe filenames:
```
firefox                → firefox.nix
python3.11            → python3-11.nix
kdePackages.marble    → kdePackages-marble.nix
python311Packages.foo → python311Packages-foo.nix
```

**Transformations:**
- `/` → `-` (path separators)
- ` ` → `_` (spaces)
- `:` → `_` (colons)

---

## Markers

### NIXORCIST_MARKER (`#$nixorcist$#`)
Identifies modules generated by nixorcist vs user-created ones.

**Used for:**
- Distinguishing managed vs manual modules in purge
- Preventing accidental deletion of custom files

### NIXORCIST-ATTRPATH
Comment storing the original package attribute path.

**Format:** `# NIXORCIST-ATTRPATH: firefox`

**Purpose:**
- Reverse lookup from module file to package
- Validation and audit trail

## Performance

- Lock file parsing: ~O(n) where n = package count
- Derivation checks: Batched via Nix to minimize eval overhead
- File I/O: Sequential write, minimal disk impact
- Safe for large lock files (100+ packages)

## Code of Conduct

- Always validate packages before generating modules
- Use `is_derivation()` not raw nix eval calls
- Preserve existing modules (check before write)
- Include markers for all generated files
- Use `show_item()` for progress feedback
- Clean up temp files on error
- Support idempotent re-runs

## Error Handling

- Non-derivations skipped with warning
- Existing modules noted but not overwritten
- Lock file read errors abort early
- Nix eval errors trapped and reported

## Dependencies
- `nix` eval command
- `cli.sh` - Visual output
- `lock.sh` - `read_lock_entries()` and `MODULES_DIR`
- GNU tools: `tr`, `mkdir`, `awk`

## Integration

Called by:
- `nixorcist gen` - Standalone
- `nixorcist all` - Part of full pipeline
- Manual calls in scripts

Depends on:
- Lock file must exist
- MODULES_DIR must be accessible
- NIXORCIST_MARKER defined in lock.sh
